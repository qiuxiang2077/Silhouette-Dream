## 🎯 概述

本文档定义了第二阶段（正式ai编码阶段）中三个AI Agent（A、B、C）的角色分工、任务目标、工作流程和协作机制。基于第一阶段（准备阶段）生成的成果，第二阶段旨在将游戏概念转化为可执行的技术规格和艺术资源，为实际开发铺平道路。

**项目背景**：《影梦》(Silhouette Dream) - 伪3D俯视角像素肉鸽冒险游戏，以「梵天一梦」式梦境逻辑为世界观基底，融合现实成长叙事与共生设定。

### 核心目标
- **Agent A**：将策划需求转化为技术规格，重点关注双重世界设定、影子共生系统、情绪投射机制
- **Agent B**：基于技术规格实现游戏系统，实现伪3D俯视角战斗、梦境-现实联动机制
- **Agent C**：创建完整的美术和音频资源，体现梦境奇幻 vs 现实写实的视觉对比

### 输入文件（来自第一阶段）
1. **`游戏设定数据库.html`** - 交互式HTML百科，包含完整的游戏设定数据库（需包含双重世界、影子共生、情绪投射等核心概念）
2. **`游戏引擎注意事项.md`** - 针对选定引擎的开发规范和最佳实践
3. **`项目策划总文档.md`** - 整合后的游戏设计文档（如有）

### 输出文件（本阶段交付物）
1. **`程序编程需求文档.md`** - Agent A 输出，需详细描述梦境-现实联动机制
2. **`Godot插件脚本包/`** - Agent B 输出，实现伪3D俯视角战斗和影子系统
3. **`美术音频资源规格书.md`** + 资源清单 - Agent C 输出，需区分梦境/现实两种美术风格

## 🤖 Agent A：程序编程需求分析师

### 角色定位
技术翻译专家，将游戏策划需求转化为精确、可执行的技术规格。

### 主要任务
1. **全面解析策划需求**
   - 分析 `game_wiki.html` 中的所有游戏元素，重点关注双重世界设定、影子共生系统、情绪投射机制
   - 理解核心游戏机制：梦境-现实联动系统、影子战斗系统、情绪化敌人AI、双重成长体系
   - 识别技术实现难点和依赖关系，特别是梦境世界随机生成与现实情绪映射的关联

2. **制定核心系统需求**
   - **梦境-现实联动系统**：睡眠/唤醒状态切换、影子出入机制、现实情绪数据导入
   - **影子战斗系统**：伪3D俯视角战斗、武器切换、技能释放、影子能力成长
   - **情绪投射敌人AI**：基于现实负面情绪的怪物生成算法、BOSS情绪映射机制
   - **双重成长系统**：局内梦境成长（临时强化）vs 局外现实成长（永久解锁）
   - **房间生成系统**：梦境关卡随机生成算法、情绪主题房间设计

3. **产出程序需求文档**
   - 详细的功能规格说明，特别强调梦境与现实的数据交互接口
   - 类/接口设计图，包括ShadowCharacter、DreamWorld、RealitySystem等核心类
   - 数据结构和算法描述，重点描述情绪-怪物映射算法
   - 性能指标和优化建议，特别是伪3D效果的性能考量

### 输入文件
- `游戏设定数据库.html`（必须）
- `项目策划总文档.md`（如有）
- `README.md`（项目概述）

### 输出文件
- `程序编程需求文档.md`（主要交付物）
- `战斗系统UML图.png`（可选）
- `技术风险评估报告.md`（可选）

## 🎨 Agent C：美术音频设计师

### 角色定位
创意资源专家，负责游戏所有视觉和听觉元素的创作。

### 主要任务
1. **美术设计**
   - **梦境世界风格**：奇幻色彩、天马行空设计、伪3D俯视角像素美术
   - **现实世界风格**：写实温暖风格、日常场景、校园家庭环境
   - **人物设计**：主角（小男孩现实形态+影子梦境形态）、NPC、情绪化敌人/Boss
   - **场景设计**：梦境房间瓦片集（奇幻主题）、现实场景（写实主题）
   - **道具武器**：图标、精灵图、动画，区分梦境武器和现实道具
   - **UI设计**：界面元素、图标、字体，区分梦境UI（奇幻）和现实UI（简洁）

2. **音频设计**
   - **背景音乐**：梦境BGM（奇幻冒险风格）、现实BGM（温馨日常风格）
   - **音效库**：攻击、受伤、交互、UI反馈，区分梦境/现实音效风格
   - **环境音效**：梦境氛围声音（奇幻）、现实环境音效（日常）

3. **资源规格制定**
   - **像素尺寸规范**：32x32像素为主，支持伪3D视角效果
   - **调色板要求**：梦境调色板（鲜艳奇幻16色）、现实调色板（温暖写实16色）
   - **动画帧数**：Idle（4帧）、Walk（8帧）、Attack（6帧）、Death（8帧）
   - **音频格式**：WAV/OGG格式，质量一致

### 输入文件
- `游戏设定数据库.html`（视觉和音频描述）
- `程序编程需求文档.md`（技术约束）
- `美术风格参考板.png`（如有）

### 输出文件
- `美术音频资源规格书.md`（资源需求和标准）
- `资源清单.csv`（所有资源的详细清单）
- 概念设计图（可选）
- 风格指南（可选）

### 资源规格要求
#### 1. 像素美术规范
- **尺寸**：角色32x32像素，场景元素16x16或32x32
- **调色板**：限制在16-32色以内，确保视觉一致性
- **动画帧数**：Idle（4帧）、Walk（8帧）、Attack（6帧）、Death（8帧）
- **文件格式**：PNG（透明背景），按精灵表（Sprite Sheet）组织

#### 2. 音频资源规范
- **背景音乐**：WAV或OGG格式，立体声，2-3分钟循环
- **音效**：WAV格式，单声道，采样率44.1kHz
- **命名规范**：`类型_名称_变体.wav`（如：`sfx_sword_swing_01.wav`）

#### 3. 资源清单格式

## 🔄 工作流程与协作

### 阶段流程
```
第一阶段成果
    ↓
Agent A 分析策划需求 → 输出程序需求文档
    ↓
Agent B 接收需求文档 → 实现插件脚本
    ↓              ↘
Agent C 接收美术需求 → 创建美术音频资源
    ↓
第二阶段成果整合
```

### 协作机制
1. **文档驱动开发**：所有需求和变更必须记录在文档中
2. **定期同步**：每完成一个模块，更新共享文档状态
3. **问题跟踪**：使用GitHub Issues或类似工具跟踪技术问题
4. **版本控制**：所有资源文件使用Git LFS管理

### GitHub版本管理流程
- **分支策略**：
  - `main`分支：稳定版本，仅接受通过测试的代码
  - `develop`分支：开发主分支，集成各功能分支
  - `feature/*`分支：新功能开发（如`feature/shadow-system`）
  - `bugfix/*`分支：Bug修复
  - `hotfix/*`分支：紧急生产环境修复
- **提交规范**：
  - `feat:`：新功能
  - `fix:`：Bug修复  
  - `docs:`：文档更新
  - `style:`：代码格式调整（不影响功能）
  - `refactor:`：代码重构
  - `test:`：测试相关
  - `chore:`：构建过程或工具更新
- **Pull Request流程**：
  1. 从`develop`分支创建功能分支
  2. 本地开发完成后推送到远程仓库
  3. 创建Pull Request到`develop`分支
  4. 至少需要一名团队成员代码审查
  5. 通过自动化测试（如果配置了CI/CD）
  6. 合并到目标分支
- **资源管理**：
  - 代码文件：使用Git标准管理
  - 美术资源（图片、音频）：使用Git LFS管理大文件
  - 二进制文件：Godot场景文件、导出资源等纳入版本控制
- **标签管理**：
  - 使用语义化版本标签（如`v0.1.0-alpha`、`v1.0.0`）
  - 每个正式发布版本创建标签

### 依赖关系管理
- Agent B 依赖 Agent A 的需求文档才能开始编码
- Agent C 可以在 Agent A 完成后开始，部分工作可与 Agent B 并行
- 美术资源可以作为占位符先行提供，后续替换为最终版本
