# GAME_MAKING - 伪3D俯视角像素肉鸽冒险游戏

## 🎮 项目概述

**游戏类型**：伪3D俯视角像素风格肉鸽冒险游戏  
**世界观基底**：「梵天一梦」式梦境逻辑，融合现实成长叙事与共生设定  
**参考游戏**：《元气骑士》《饥荒》《喵喵启示录》《星露谷物语》《杀戮尖塔》《土豆兄弟》 
**核心循环**：梦境冒险（局内随机成长） + 现实成长（局外永久解锁）  
**美术风格**：伪3D俯视角像素风，梦境奇幻 vs 现实写实  
**游戏引擎**：Godot 4.x  
**开发状态**：概念设计阶段  
**暂定标题**：《影梦奇旅》

### 世界观概述
- **双重世界**：表层是剑与魔法+科幻融合的梦境冒险世界；底层真相是爱幻想小男孩的梦，影子是共生种族
- **现实-梦境联动**：小男孩入睡后影子进入梦境战斗，怪物是现实经历/情绪的投射
- **核心主题**：内心成长与自我和解，通过梦境冒险反映现实心理蜕变

### 核心玩法特色
- **梦境冒险**：伪3D俯视角肉鸽战斗，随机生成梦境关卡，局内拾取武器/道具/技能
- **现实叙事**：小男孩日常互动（校园、家庭、情绪变化），影响梦境内容和难度
- **影子共生**：影子作为主角在梦境中的化身，能力随现实成长而进化
- **情绪投射**：怪物和BOSS是现实负面情绪（如霸凌）的具象化
- **双结局系统**：普通结局（完成冒险） + 隐藏真结局（揭露世界真相）

## 🗓️ 16周开发规划

### 阶段1：概念设计（第1-2周）
- 确定故事世界观（一页纸梗概）
- 创建美术风格参考板
- 建立开发环境与Git仓库

### 阶段2：预生产（第3-4周）
- 玩家角色精灵与移动系统
- 基础房间瓦片集与摄像机跟随
- 第一个武器（近战剑）与基础敌人（史莱姆）

### 阶段3：核心系统开发（第5-8周）
- 武器系统基类与切换
- 敌人AI状态机（巡逻→追击→攻击）
- 房间生成系统（随机连接3-5个房间）
- UI框架与音效系统

### 阶段4：内容生产（第9-12周）
- 制作5-8种敌人（含1个Boss）
- 制作10-15种武器/技能
- 实现局外成长与商店系统

### 阶段5：测试与优化（第13-15周）
- 平衡性测试与数值调整
- Bug排查与性能优化（目标60FPS稳定）
- 小范围玩家测试收集反馈

### 阶段6：发布准备（第16周）
- 多平台打包（Windows/macOS/Linux）
- 商店素材准备（截图、图标、封面图）
- 最终测试与版本归档

## 🔧 技术实现要点

### 引擎与工具
- **主引擎**：Godot 4.x（2D引擎对像素游戏支持优秀）
- **美术工具**：Aseprite / Pixelorama
- **版本控制**：Git + GitHub
- **项目管理**：Trello看板 + 飞书文档

### 关键实现设置
1. **像素美术**：纹理导入Filter → Nearest，关闭Mipmaps，启用像素对齐
2. **场景结构**：节点化设计，分离UI层、世界层、实体层
3. **武器系统**：基类继承模式，支持热插拔切换
4. **数据持久化**：Resource格式保存永久进度

### 项目目录结构

## 🤝 协作与沟通

### 协作流程
```
核心团队分析需求 → 填写需求文档 → 讨论确认细节 → 外部同学生产 → 验收集成
```

### 需求文档模板（简化版）
- **美术需求**：ART_REQUIREMENT_TEMPLATE.md
- **音乐需求**：AUDIO_REQUIREMENT_TEMPLATE.md

### 沟通建议
- **例会频率**：每周一次（周日晚上，1小时）
- **沟通工具**：微信群（日常）+ 飞书文档（正式文档）
- **进度同步**：每3-4天更新进度，遇到问题尽早沟通

### GitHub版本管理流程
- **仓库结构**：主分支`main`用于稳定版本，开发分支`develop`用于日常开发，功能分支`feature/*`用于新功能开发
- **提交规范**：遵循Conventional Commits格式（feat:, fix:, chore:, docs:, style:, refactor:, test:）
- **分支策略**：
  - `feature/`：新功能开发，从`develop`分支创建，完成后合并回`develop`
  - `bugfix/`：Bug修复，从`develop`分支创建
  - `hotfix/`：紧急修复，从`main`分支创建，修复后同时合并到`main`和`develop`
- **Pull Request流程**：
  1. 开发者在本地完成功能开发并推送到远程分支
  2. 创建Pull Request到`develop`分支
  3. 至少需要一名团队成员代码审查通过
  4. 通过CI/CD流水线测试
  5. 合并到目标分支
- **标签管理**：使用语义化版本标签（v1.0.0）标记发布版本
- **GitHub Actions**：配置自动化测试、构建和部署流水线
